## 1. Design Proposal

### Round selector: scrollable chip strip

A horizontal scrollable row of MUI `Chip` components, visually distinct from the existing top-level `Tabs` (Schedule / Live Match / Track Record). The top tabs use underline indicators and text-only labels; the round chips use filled/outlined variant toggling with the `homeAccent` orange for the selected chip. This avoids two rows of tabs that look identical and creates a clear visual hierarchy: tabs for page sections, chips for round filtering within the Schedule.

A dropdown was considered but hides all options behind a click — for ~9 items the user should see and scan the full list without interaction. Scrollable tabs were rejected because they're visually indistinguishable from the existing top tabs.

### Round labels

**Flag raised:** The prompt said "Round labels: API-Football strings verbatim" and also "~9 rounds (3 group matchdays + R32 + R16 + QF + SF + 3rd + Final)". I initially assumed these conflicted (per-group strings like "Group A - 1" would give ~42 items). After deploying and checking the actual API-Football data, WC 2026 returns **already-aggregated** strings: `"Group Stage - 1"`, `"Group Stage - 2"`, `"Group Stage - 3"`. Only 3 unique group-stage labels, not 36.

**Resolution:** Round labels are used verbatim from API-Football. The grouping function also handles the per-group format ("Group A - 1" → "Group Stage - 1") as a robustness fallback, in case API-Football changes the format once the tournament starts.

### Date filter: scrollable date chips

Below the round selector, a second chip strip shows the unique match dates within the selected round, plus an "All" chip at the left. When "All" is selected, fixtures display grouped by date with sticky day headers (existing `FixtureList` behavior). When a specific date is selected, only that day's fixtures render.

### Default-selection rule

1. **Round selection:** Find the first round category (in tournament order) that has at least one fixture whose kickoff date falls on today (user's local timezone). If found, select it. If no round has matches today, select the first round whose earliest fixture kickoff is in the future. If all rounds are in the past, select the last round.

2. **Date selection:** If the selected round has matches today, default the date chip to today. Otherwise, default to "All".

3. **Timezone:** All comparisons use the user's local timezone via `new Date()`. The user sees dates in their own timezone, consistent with the existing `FixtureList` behavior.

4. **No remembered state:** Navigation away and back resets to the default selection. No `localStorage`, no URL params.

### Empty state

When the selected round + date combination has zero fixtures:

> No matches scheduled for this date in [round label].

Centered, grey text (`#b0bec5`), same typography as the existing empty state.

## 2. Before-state

### SchedulePage

A thin wrapper: receives all fixtures from outlet context, passes them directly to `FixtureList`. No round awareness, no filtering.

### FixtureList

Groups all fixtures by local date with sticky day headers. No round grouping. Shows every fixture for the entire tournament in one flat list.

### Backend

No `/fixtures/rounds` endpoint. Round info existed only inside each fixture's `league.round` field.

## 3. After-state

### Backend: `/fixtures/rounds` route

`backend/football/routes.py`:

```python
@router.get("/fixtures/rounds")
async def list_rounds(
    response: Response,
    league: int = WC_LEAGUE_ID,
    season: int = WC_SEASON,
    client: APIFootballClient = Depends(get_football_client),
) -> dict:
    try:
        rounds = await client.get_rounds(league=league, season=season)
    except APIFootballError as exc:
        _raise_for_football_error(exc)

    _set_cache(response, _CC_FIXTURES_LIST)
    return {
        "count": len(rounds),
        "rounds": rounds,
    }
```

`backend/football/data_provider.py` — new method:

```python
async def get_rounds(
    self,
    league: int = WC_LEAGUE_ID,
    season: int = WC_SEASON,
) -> list[str]:
    items = await self._request(
        "/fixtures/rounds",
        {"league": league, "season": season},
        cache_method="rounds",
        ttl=ENDPOINT_TTLS["rounds"],
    )
    return [str(r) for r in items]
```

`backend/football/constants.py` — added `"rounds": 3_600` to `ENDPOINT_TTLS`.

### Frontend: round-grouping utility

`frontend/src/football/utils/roundGrouping.ts`:

- `getRoundCategory(round)` — passes verbatim API-Football strings through; collapses per-group format ("Group A - 1" → "Group Stage - 1") as fallback
- `groupFixturesByRound(fixtures)` — groups and sorts by tournament order
- `getRoundCategories(fixtures)` — extracts sorted unique categories
- `getDefaultRound(roundGroups)` — today's round → next upcoming → last
- `getDefaultDate(fixtures)` — today if matches today, else "all"
- `getUniqueDates(fixtures)`, `toLocalDateKey()`, `formatDateChip()`, `getTodayKey()`

### Frontend: RoundSelector component

`frontend/src/football/components/RoundSelector.tsx`:

Scrollable `Box` with `Chip` per round. Selected chip: filled orange (`#ff6f00`), bold. Unselected: outlined, grey text. `data-testid="round-selector"` and `data-testid="round-chip"`.

### Frontend: DateFilter component

`frontend/src/football/components/DateFilter.tsx`:

Same pattern as RoundSelector but smaller chips. "All" chip first, then date chips formatted as "Jun 11". `data-testid="date-filter"` and `data-testid="date-chip"`.

### Frontend: refactored SchedulePage

`frontend/src/pages/football/SchedulePage.tsx`:

1. Computes `roundGroups`, `roundCategories`, `defaultRound` from fixtures via `useMemo`
2. `selectedRound` state initialized from `defaultRound` via `useEffect`
3. `selectedDate` state resets when round changes (via `useEffect` on `roundFixtures`)
4. Filters fixtures by round then date
5. Renders `RoundSelector` → `DateFilter` → `FixtureList` (or empty state)
6. Loading, error, and global empty states unchanged

### Unchanged

- `FixtureList` — no modifications
- `FixtureCard` — no modifications
- `WorldCup2026Layout` — no modifications
- `useFixtures` hook — no modifications
- All fixture data flows through the existing outlet context

## 7. Surprises

### API-Football round format

The prompt assumed per-group round strings ("Group A - 1", etc.) giving ~42 items. The actual API-Football data for WC 2026 returns already-aggregated strings: "Group Stage - 1", "Group Stage - 2", "Group Stage - 3". This means:

1. The verbatim-vs-aggregation tension raised in the design proposal was a non-issue — verbatim strings ARE the aggregated format
2. The per-group fallback in `getRoundCategory` is unnecessary for WC 2026 but preserved for robustness in case the format changes

### No knockout rounds yet

Only group-stage rounds are currently returned (72 fixtures, 3 rounds). Knockout rounds ("Round of 32", "Round of 16", etc.) will appear when API-Football publishes them. The `ROUND_ORDER` constant in the frontend handles this — new rounds will sort correctly when they appear.

---

## 8. Open questions

1. **Knockout round label format.** The `ROUND_ORDER` array assumes strings like "Round of 32", "Quarter-finals", etc. These are based on API-Football documentation but haven't been verified against live WC 2026 data (since the bracket doesn't exist yet). If the labels differ slightly, only `ROUND_ORDER` in `roundGrouping.ts` needs updating — the sort falls back to alphabetical for unrecognized strings.

2. **Per-group fallback.** The `getRoundCategory` function collapses `"Group A - 1"` → `"Group Stage - 1"`. This was designed before seeing the actual data. If API-Football ever switches to per-group strings mid-tournament, the fallback handles it. If not, it's dead code that costs nothing.
