## 1. Design Proposal

### Route & navigation placement

`/football/world-cup-2026/standings` — nested inside `WorldCup2026Layout` as a fourth tab alongside Schedule, Live Match, and Track Record. This keeps all WC 2026 content under the same layout (shared header, fixture context, tab bar) and avoids a separate standalone page that would need its own Header/layout treatment.

The Header.tsx Football dropdown also gets a "Standings" MenuItem for direct access.

### Backend: `/standings` endpoint

`GET /api/football/standings?league=1&season=2026`

Response shape mirrors API-Football's `/standings` endpoint, which returns a nested structure: `response[0].league.standings` is an array of arrays — one inner array per group. Each inner array contains 3–4 team objects (for WC 2026: 4 teams per group × 12 groups = 48 teams).

Pydantic models:
- `AFStandingTeam` — `{id, name, logo}`
- `AFStandingStats` — `{played, win, draw, lose, goals: {for, against}}`
- `AFStandingEntry` — `{rank, team, points, goalsDiff, group, form, status, description, all: AFStandingStats, home, away, update}`
- `AFStandingsLeague` — `{id, name, country, logo, flag, season, standings: list[list[AFStandingEntry]]}`
- `AFStandingsResponse` — `{league: AFStandingsLeague}`

Server TTL: 10 minutes (`ENDPOINT_TTLS["standings"] = 600`).
CDN TTL: 60 seconds (`_CC_STANDINGS = 60`).
Stale fallback: enabled (default).

### Frontend: StandingsPage component

**Group card structure:**

Each group is a `Card` containing:
1. **Header** — Group name (e.g., "Group A") in bold, left-aligned, with a subtle orange left-border accent
2. **Table** — `<table>` with columns: Pos, Team, P, W, D, L, GF, GA, GD, Pts

**Team cell:** CSS flag icon (via `flagClass()`) + team name. Falls back to API logo if no flag mapping.

**Responsive column collapse:**
- Desktop (≥900px): All 10 columns visible — Pos, Team, P, W, D, L, GF, GA, GD, Pts
- Tablet (600–899px): Hide GF, GA — show Pos, Team, P, W, D, L, GD, Pts (8 columns)
- Mobile (<600px): Minimum — Pos, Team, P, GD, Pts (5 columns)

Column hiding uses `display: { xs: 'none', sm: 'table-cell' }` and `display: { xs: 'none', md: 'table-cell' }` MUI sx breakpoint syntax.

**Grid layout:**
- Mobile: 1 column (`xs=12`)
- Tablet (≥600px): 2 columns (`sm=6`)
- Desktop (≥900px): 2 columns (`md=6`) — 12 groups × 2 col = 6 rows. 3-column wastes space with the table width.

**Pts column:** Bold, orange text (`#ff6f00`) for emphasis.

**Loading state:** 6 `Skeleton` cards in a 2-column grid, each with a card header skeleton + 4 row table skeleton.

**Error state:** Same pattern as SchedulePage — centered error text with Retry button, `data-testid="standings-error"`.

**Empty state:** "Standings will be available once the group stage begins." centered grey text, `data-testid="standings-empty"`.

### Backend: `get_standings()` method

`backend/football/data_provider.py`:

```python
async def get_standings(
    self,
    league: int = WC_LEAGUE_ID,
    season: int = WC_SEASON,
) -> AFStandingsResponse | None:
    items = await self._request(
        "/standings",
        {"league": league, "season": season},
        cache_method="standings",
        ttl=ENDPOINT_TTLS["standings"],
    )
    if not items:
        return None
    return AFStandingsResponse.model_validate(items[0])
```

### Backend: `/standings` route

`backend/football/routes.py`:

```python
@router.get("/standings")
async def get_standings(
    response: Response,
    league: int = WC_LEAGUE_ID,
    season: int = WC_SEASON,
    client: APIFootballClient = Depends(get_football_client),
) -> dict:
    try:
        standings = await client.get_standings(league=league, season=season)
    except APIFootballError as exc:
        _raise_for_football_error(exc)
    if standings is None:
        _set_cache(response, _CC_STANDINGS)
        return {"league": None, "groups": []}
    _set_cache(response, _CC_STANDINGS)
    return standings.model_dump(mode="json", by_alias=True)
```

CDN TTL: `_CC_STANDINGS = 60` (60 seconds).

## 7. Surprises

### 13 groups instead of 12

API-Football returns 13 groups for WC 2026 instead of the expected 12. The WC 2026 expanded format includes 48 teams across 12 groups of 4, but the API-Football data currently shows a 13th entry. This may be a play-in bracket placeholder or an API-Football artifact. The StandingsPage renders whatever groups the API returns — no hardcoded count — so this handles itself.

### All stats null pre-tournament

API-Football returns `null` for every integer stat field (played, win, draw, lose, goals.for, goals.against, points, goalsDiff) before the tournament starts. The first deployment returned 500 because the Pydantic models expected `int` fields. Fixed by adding `model_validator(mode="before")` to coerce `None → 0` on `AFStandingGoals`, `AFStandingStats`, and `AFStandingEntry`. This is safe because 0 is the correct value for pre-tournament stats.

## 8. Open questions

1. **13th group.** API-Football returns 13 group arrays. If this is a temporary artifact that resolves to 12 before June 11, no action needed. If it persists, the page renders 13 cards correctly — just an odd number in a 2-column grid (last row has 1 card).

2. **Standings during knockout.** The endpoint returns group standings only. Once knockout rounds begin, API-Football may still return group standings (frozen final state) or stop returning data. The StandingsPage handles both — empty response shows the empty state, frozen data shows the final group tables.
